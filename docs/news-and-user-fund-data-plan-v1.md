# 实时新闻与用户自选基金数据接入方案 V1

## 1. 背景

当前产品已经具备行业机会、基金发现、基金对比和观察列表的前端演示闭环，后端也已经具备 PostgreSQL、FastAPI、AKShare 盘后采集和长期持有成本快照的基础能力。

下一阶段需要解决两个用户感知最强的问题：

- 用户希望随时看到重要财经时事新闻，并能关联到行业观察。
- 用户希望搜索自己真实持有或关注的基金，添加后再拉取真实数据，而不是只能看预设候选池。

本方案仍以个人基金筛选和长期观察为核心，不把新闻或短期热度直接表达成买卖建议。

## 2. 金十新闻接入原则

### 2.1 接入边界

金十可以作为重要实时财经新闻来源接入，但前提是使用官方授权 API、商务接口或其他明确许可的数据服务。

不建议采用网页逆向、非授权抓取或绕过登录态的方式，因为这会带来稳定性、合规和版权风险。

### 2.2 产品定位

金十新闻在本产品中定位为“事件线索”和“观察提醒”，不作为基金推荐依据。

适合落地到以下位置：

- 首页：重要快讯栏，展示高优先级宏观、政策、行业事件。
- 行业详情页：事件时间线，展示与当前行业相关的新闻。
- 我的观察页：最近变化摘要，提示用户关注的行业或基金是否有新事件。

### 2.3 后端建议

建议新增独立新闻域，不混入基金日度指标表。

建议表：

- `news_event_raw`：保存原始新闻事件、来源、发布时间、外部 ID、抓取批次。
- `news_event_normalized`：保存标准化后的标题、摘要、重要性、标签、相关资产类型。
- `industry_news_mapping`：保存新闻与行业主题的映射关系。

建议任务：

- `news_event_fetch`：按授权 API 拉取最近新闻，支持分页、限速、重试和去重。
- `news_event_classify`：按关键词和行业词典做轻量映射，后续可升级为模型分类。
- `news_snapshot_publish`：生成首页和观察页可直接消费的新闻快照。

建议 API：

- `GET /api/news/important`：返回首页重要新闻。
- `GET /api/industries/{industry_id}`：在行业详情快照中附带相关新闻。
- `GET /api/watchlist-summary`：在观察摘要中附带关注项相关事件。

## 3. 用户搜索添加基金的数据链路

### 3.1 当前问题

当前真实数据链路只会采集 `backend/templates/coverage/fund_universe.csv` 里已经配置的基金。页面中额外展示的部分候选基金来自前端演示样例，因此看起来不像真实基金。

这不是前端展示问题，本质是“基金候选池还没有用户自助扩展能力”。

### 3.2 目标体验

用户在页面输入基金代码或基金名称，例如：

- 嘉实全球产业升级
- 南方北证50
- 159 开头或 5 开头的 ETF 代码
- 场外公募基金代码

系统返回可选基金列表。用户选择其中一个基金并添加到观察池或候选池后，后端把它加入采集队列，盘后或手动触发真实数据拉取。

### 3.3 推荐链路

1. 前端搜索基金。
2. 后端通过 AKShare 基金目录或本地已缓存基金目录查询候选项。
3. 用户确认添加。
4. 后端写入 `fund_user_candidate` 或扩展后的候选池表。
5. 采集任务拉取基金主数据、净值历史、风险收益指标。
6. 费用规则先用可获得数据填充，缺失部分标记为 `partial`。
7. 页面展示该基金，并明确数据状态：
   - `待采集`
   - `采集中`
   - `真实快照`
   - `费用规则待补充`
   - `采集失败，可重试`

### 3.4 数据表建议

新增用户自选候选域：

- `fund_candidate_request`
  - `request_id`
  - `fund_code`
  - `fund_name_query`
  - `matched_fund_name`
  - `source_name`
  - `request_status`
  - `created_at`
  - `updated_at`

- `fund_collection_task`
  - `task_id`
  - `fund_id`
  - `fund_code`
  - `task_type`
  - `task_status`
  - `attempt_count`
  - `last_error_message`
  - `created_at`
  - `updated_at`

也可以先不引入复杂用户系统，把“用户添加”理解为本机候选池维护，后续再加登录和多用户隔离。

### 3.5 API 建议

- `GET /api/funds/search?q=嘉实全球产业升级`
  - 查询 AKShare 目录或本地基金目录缓存。
  - 返回基金代码、基金名称、基金类型、拼音缩写、数据来源。

- `POST /api/funds/candidates`
  - 用户确认添加基金。
  - 写入候选池。
  - 返回采集状态。

- `POST /api/funds/{fund_code}/collect`
  - 手动触发单只基金采集。
  - 开发阶段可同步执行，后续改成后台任务。

- `GET /api/funds/{fund_code}/collection-status`
  - 返回当前基金采集状态、最后成功日期和失败原因。

## 4. AKShare 采集注意事项

AKShare 可以支撑当前阶段的主数据和日度净值/行情采集，但需要继续保持低频、分批、可重试。

默认策略：

- 搜索和目录缓存可以相对频繁，但仍应设置本地缓存。
- 单只基金添加后优先拉主数据，再拉日度历史。
- 批量采集要保留随机等待、分块冷却和失败重试。
- QDII、ETF、场外主动基金的数据接口可能不同，失败时必须保留可解释状态。
- 费用和赎回费规则不一定能完整自动获取，第一版允许“真实净值 + 部分费用规则”的混合状态。

## 5. 与产品指标体系的关系

用户添加真实基金后，页面应优先补齐对个人选基更有意义的指标：

- 基金名称与代码
- 基金类型
- 基金公司
- 成立年限
- 基金规模
- 近 1/3/6/12 月收益
- 最大回撤
- 波动率
- 跟踪标的或投资主题
- 申购费、管理费、托管费、销售服务费
- 赎回费规则和多少天免赎回费

行业热度、新闻事件和 PE/PB 仍作为研究辅助，不直接替代基金筛选结论。

## 6. 下一轮实施建议

建议按以下顺序推进：

1. 新增基金搜索 API：先基于 AKShare 目录或缓存实现基金代码/名称搜索。
2. 新增添加候选基金 API：把用户选择的基金写入候选池。
3. 新增单只基金采集任务：支持用户添加后手动触发拉取。
4. 前端基金发现页增加搜索添加入口和采集状态。
5. 接入金十新闻授权配置和新闻表结构。
6. 首页、行业详情页和观察页展示重要新闻摘要。

这样可以先解决“我自己的基金能不能进来”的核心体验，再逐步增强新闻事件层。

## 7. 长期行业事件分析升级

第一版行业资讯不追求实时抢新闻，而是服务盘后长期趋势复盘。事件进入系统后必须被标记为：

- 长期 thesis 支撑。
- 风险/失效信号。
- 短期扰动。
- 多空混合。
- 证据不足。

允许来源：

- 授权 API。
- 手动导入。
- 内部维护标签。
- Mock 样例。

默认不采用非授权网页抓取、绕过登录态或版权边界不清晰的方式。若未来接入自动资讯源，必须先补充授权说明、限速、去重、来源审计和失败降级策略。

事件字段至少包括：

- 来源和来源类型。
- 发布时间和事件日期。
- 行业映射。
- 事件类别。
- 长期影响标签。
- 置信度。
- 新鲜度。
- thesis 影响说明。
- 风险提示。
- 失效信号。

CPO 光通信作为 AI 算力的重要通信链条，第一版需要覆盖 AI 需求、光模块供应链、资本开支周期、海外需求、估值和过热风险。事件只用于强化或削弱长期观察逻辑，不直接作为买入、卖出或加仓依据。
