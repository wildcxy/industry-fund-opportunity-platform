# 基金数据源与稳定采集策略 V1

## 结论

现阶段不建议立刻购买 Tushare Pro。我们的产品核心是“个人基金持仓、选基对比、盘后决策辅助”，不是实时交易系统；第一阶段应该采用“天天基金/东方财富口径优先 + AKShare 封装调用 + 本地缓存与重试 + 可替换 Provider”的路线。

## 数据源分工

### AKShare

AKShare 适合作为第一阶段采集入口，因为它已经封装了天天基金、东方财富等公开接口，开发成本低，适合快速跑通：

- 基金目录搜索
- 基金基础信息
- 开放式基金净值走势
- ETF 行情
- 基金费率
- 天天基金/东方财富 F10 披露持仓

风险是 AKShare 本身不是商业 SLA 数据源，接口字段和上游页面结构可能变化，因此不能把它当成唯一不可替换底座。

### 天天基金 / 东方财富

天天基金/东方财富更接近我们真正需要的基金数据口径，尤其适合：

- 基金净值
- 基金档案
- 费率规则
- 持仓披露
- 基金规模
- 基金经理与历史业绩

建议下一阶段把“天天基金/东方财富”抽象成独立 Provider，即使底层仍通过 AKShare 调用，也要在代码里明确它是主数据源口径。后续如果 AKShare 不稳定，可以直接替换为自写 HTTP 采集器。

### Tushare

Tushare 更适合补充 A 股市场、指数、行业、股票基础数据。对“纯基金盘后净值与持仓对比”来说，它不是第一优先级。

适合未来使用 Tushare 的场景：

- 需要稳定的股票、指数、行业行情与估值数据
- 需要把基金持仓股票映射到 PE/PB、行业、主题暴露
- 需要做“基金持仓穿透后的行业估值与风格暴露”
- 需要比公开页面更稳定的结构化市场数据
- 需要用申万行业、指数行情、行业估值来增强“行业基金机会捕捉”
- 需要把军工、商业卫星、航天、芯片存储、新能源等新主题映射到更稳定的指数或行业口径

不建议现在就依赖 Tushare 的场景：

- 只做基金净值更新
- 只做支付宝持仓截图导入
- 只做基金费率、规模、披露持仓展示

### Tushare 接入状态

当前代码已经预留 `ProProvider`，并在 `backend/.env` 中增加：

- `TUSHARE_ENABLE`
- `TUSHARE_TOKEN`
- `TUSHARE_REQUEST_TIMEOUT_SECONDS`
- `TUSHARE_RETRY_COUNT`
- `TUSHARE_RETRY_BACKOFF_SECONDS`

默认不开启 Tushare，避免没有 token 或积分权限时影响当前 AKShare/天天基金主链路。后续启用时，建议先用于行业和指数增强，再逐步用于基金持仓穿透。

优先接入顺序：

1. `index_classify`：建立申万行业分类与行业 ID 映射。
2. `index_daily`：补充行业/主题指数日线，计算 5/20/60 日涨跌和趋势强度。
3. `fund_portfolio`：补充基金持仓穿透，估算行业暴露。
4. `fund_nav`：作为 AKShare/天天基金净值的校验源，而不是第一天就替代主链路。
5. `fund_daily`：主要用于 ETF 场内交易行情，需要积分更高，后续再评估。

积分门槛提示：Tushare 官方文档显示，基金基础信息、基金净值、基金持仓和申万行业分类通常需要至少 2000 积分；ETF 日线通常需要更高积分。最终可用接口以你的账号权限为准。

## 稳定采集策略

### 调度

盘后每日执行一次即可，建议 Windows Task Scheduler 默认在 21:30 之后执行，QDII 可增加次日补采。

推荐顺序：

1. 同步基金候选池
2. 拉取基金基础信息
3. 拉取净值与收益指标
4. 拉取费率与赎回规则
5. 拉取披露持仓
6. 计算持仓收益与组合画像
7. 写入快照表
8. 生成数据质量报告

### 重试

当前后端已有 AKShare 请求重试参数：

- `akshare_sleep_min_seconds`
- `akshare_sleep_max_seconds`
- `akshare_retry_count`
- `akshare_retry_backoff_seconds`
- `akshare_chunk_size`
- `akshare_chunk_cooldown_seconds`

建议保持“小批量、慢速、失败可重跑”：

- 单只基金失败不影响整批任务
- 每只基金记录采集任务状态、失败原因、尝试次数
- 对 QDII、费率、持仓披露这类易失败数据允许局部缺失
- 页面必须显示数据质量，不允许用演示值覆盖真实缺口

### 缓存

建议按交易日缓存原始响应：

- `runtime/data-archive/raw/<trade-date>/fund_nav/`
- `runtime/data-archive/raw/<trade-date>/fund_fee/`
- `runtime/data-archive/raw/<trade-date>/fund_holding/`

这样当天接口失败时，可以用前一次成功结果继续构建页面，并标记“非当日最新”。

## 决策辅助系统演进

个人持仓导入后，决策辅助系统应分三层，不直接输出买卖指令：

### 组合体检

- 单一主题集中度
- QDII 占比
- 科技/AI/半导体/北证/资源等主题暴露
- 单只基金持仓金额占比
- 浮盈浮亏分布

### 基金质量筛选

- 长期持有成本
- 回撤与波动
- 规模与成立年限
- 跟踪误差或主动基金风格稳定性
- 基金是否真正代表目标行业

### 操作辅助提示

- “继续观察”
- “数据不足”
- “主题过度集中”
- “短期涨幅过快，避免只因热度追高”
- “回撤较深，需要确认行业基本面和基金风格是否变化”

所有提示都必须是决策辅助，不应写成“买入/卖出/必涨”。

## 下一阶段建议

1. 先把支付宝截图导入做成“图片上传 + OCR 草稿 + 人工确认 + 本地持仓簿”。
2. 再把天天基金/东方财富 Provider 从 AKShare 里显式抽象出来。
3. 然后补盘后任务的原始响应缓存、失败重试和质量报告。
4. 最后再评估是否购买 Tushare Pro，用于持仓穿透、行业估值和指数增强，而不是替代基金净值主链路。
