## 2. 阶段结论

基于当前 PRD V1、前端架构方案和现有工程状态，本项目现阶段必须严格按照 PRD 文档先完成前端演示版建设，不进入后端开发，不提前扩展真实数据接入、账户体系、交易能力、个性化推荐引擎及其他超出 PRD V1 边界的能力。

当前阶段的唯一主线是：围绕 PRD 中定义的五大页面、核心用户链路、指标口径、风险提示和 Mock 数据展示能力，完成一个可评审、可演示、可继续迭代的前端产品版本。后端开发不是当前阶段目标，只有在前端范围按 PRD 验收完成后，才允许进入下一阶段的后端设计与开发。

## 3. 当前阶段必须遵守的实施原则

### 3.1 严格以 PRD 为唯一范围基线

1. 当前开发范围必须以 `prd-v1-industry-fund-opportunity.md` 为准。
2. 若出现“看起来有价值但 PRD 未定义”的功能，默认不进入本阶段。
3. 前端页面、模块命名、指标口径、文案表达和交互链路必须与 PRD 保持一致。
4. 任何新增能力若会改变“发现机会 - 验证行业 - 筛选基金 - 对比 - 加入观察”的主链路，应先更新 PRD 再开发。

### 3.2 当前阶段只做前端，不做后端

1. 允许继续建设前端页面、前端组件、前端状态管理、Mock 数据、前端本地 API Route 和演示用本地持久化。
2. 不允许在本阶段启动真实后端服务、数据库建模、用户系统、交易接口、实时行情接入或后台管理系统开发。
3. 现有 `app/api/*` 路由仅视为前端演示层的 Mock API 组织方式，不视为后端开发启动。
4. 所有数据交互目前仅服务于前端演示，不承诺真实数据源、不设计生产级后端协议。

### 3.3 架构目标服从演示完整性

1. 当前架构优先保证前端演示闭环完整，而不是提前引入复杂工程设施。
2. 所有技术实现应服务于 PRD 指定页面和用户路径，不得偏离产品展示目标。
3. 必须保留未来接入真实数据的扩展点，但不能为了未来可能需求而牺牲当前开发效率和清晰度。

### 4.2 核心页面骨架已完成

## 5. 前端剩余开发项

当前工程已具备演示版基础，但距离“严格按 PRD 完成前端”仍有明确差距。后续必须优先完成以下前端事项，不得跳转到后端开发。

### 5.1 页面完整性补齐

1. 对照 PRD 补齐首页的全部模块结构，包括市场概览、行业机会榜单、行业热力图、重点行业卡片、热门基金映射、风险提示等区块的完整口径与布局。
2. 对照 PRD 补齐行业详情页的信息层级，包括一句话结论、机会结论卡、多维指标卡、趋势图、资金图、催化事件时间线、相关基金列表和风险说明。
3. 对照 PRD 补齐基金发现页的筛选区、排序区、结果摘要、卡片视图与表格视图切换、空状态和异常状态。
4. 对照 PRD 补齐基金对比页的指标对比表、收益风险摘要、持仓与风格摘要、少于 2 只基金时的引导态。
5. 对照 PRD 补齐我的观察页的行业观察、基金观察、最近变化摘要和空状态引导。

### 5.2 交互链路补齐

1. 完成从首页进入行业详情页的链路校验与展示一致性。
2. 完成从行业详情页进入基金发现页时带入行业筛选条件的链路。
3. 完成从基金发现页加入对比池并进入基金对比页的链路。
4. 完成从各页面加入观察后在观察页可见的链路。
5. 完成路由参数、页面状态与本地持久化状态之间的一致性。

### 5.3 数据与类型层补齐

1. 将当前类型定义完全对齐 PRD 中要求的 `IndustryOpportunityCard`、`IndustryDetailSnapshot`、`FundListItem`、`FundCompareItem`、`WatchlistItem`。
2. 将 Mock 数据从“可展示”提升为“与 PRD 字段和场景完全一致”的结构化数据。
3. 建立页面模型或适配层，避免页面直接耦合原始 Mock 数据。
4. 为指标、标签、风险提示补齐口径说明字段与展示位。

### 5.4 视觉与信息表达补齐

1. 修正当前页面中尚未完成的正式中文展示内容，确保所有前端界面符合正式金融产品表达。
2. 统一标签、分值、风险等级、口径说明和免责声明的视觉规范。
3. 确保所有模块都体现“可解释性优先”和“非投资建议”的合规边界。
4. 完成桌面端优先且兼顾移动端响应式适配。

### 5.5 质量与验收能力补齐

1. 增加页面级空状态、异常状态和无数据状态。
2. 增加基础类型校验、页面联调检查和关键交互测试。
3. 对照 PRD 补齐所有关键模块后的自测清单。
4. 形成可供产品、设计、架构与开发共同评审的前端演示版本。

## 6. 前端完成的验收口径

当前阶段何时可认定“前端已完成”，必须以 PRD 为准，而不是以“页面能打开”或“已有部分静态展示”为准。前端完成的验收口径如下：

### 6.1 范围验收

1. PRD 定义的五大页面全部完成，且页面目标、模块、关键字段、交互动作、异常状态和空状态均已落地。
2. 完整覆盖用户主链路：发现机会 -> 验证行业 -> 筛选基金 -> 对比 -> 加入观察。
3. 所有页面使用统一术语和统一指标口径，不出现与 PRD 冲突的命名和表达。

### 6.2 交互验收

1. 首页可进入行业详情页。
2. 行业详情页可跳转基金发现页并保留行业筛选上下文。
3. 基金发现页可进行筛选、排序、加入对比与加入观察操作。
4. 基金对比页可正确展示 2 至 4 只基金的横向对比。
5. 我的观察页可稳定展示已加入的行业与基金，并支持移除与跳转。

### 6.3 数据与展示验收

1. 页面展示字段与 PRD 数据契约一致。
2. Mock 数据足以覆盖榜单、详情、筛选、对比、观察、风险提示等核心场景。
3. 评分、标签、排序和风险提示均附带说明位，不出现黑盒式展示。
4. 不出现“稳赚”“抄底”“确定性收益”等偏离 PRD 的违规表述。

### 6.4 工程验收

1. 前端工程可稳定运行、构建和进行基本类型检查。
2. 路由清晰，页面边界明确，基础组件和业务模块职责可区分。
3. Mock API、页面数据访问和本地持久化行为稳定可复现。
4. 主要页面在桌面端和移动端均能完成基础浏览与关键操作。

## 7. 何时再进入后端

后端开发必须在前端阶段满足以下条件后才可启动：

1. PRD V1 范围内的前端五大页面已全部按验收口径完成。
2. 产品、设计、前端和架构四方对当前前端版本完成联合评审，并确认不再新增 V1 前端范围内的重大结构性变更。
3. Mock 数据结构、页面数据契约和前端适配层已经稳定，足以作为后端接口设计输入。
4. 已完成一轮基于前端成品的链路走查，确认“发现机会 - 验证行业 - 筛选基金 - 对比 - 加入观察”闭环顺畅。
5. 已单独输出下一阶段后端目标说明，明确后端仅服务于前端既定页面，而不是重新定义产品范围。

在上述条件满足之前，不建议、不允许启动真实后端开发。若提前进入后端，会导致接口设计反复、页面需求返工和前后端边界失稳，不符合当前项目推进顺序。

## 8. 下一阶段前端开发建议顺序

为确保严格按照 PRD 推进，建议按以下顺序继续前端开发：

1. 先补齐页面信息架构与正式中文内容，确保所有页面模块与 PRD 对齐。
2. 再补齐基金发现页、基金对比页、我的观察页的真实前端交互闭环。
3. 然后统一数据契约、Mock 数据、适配层和本地状态管理。
4. 最后补齐空状态、异常状态、响应式适配和基础测试。

## 9. 架构师结论

当前项目仍处于“前端演示版建设阶段”。从架构角度看，现有工程已经完成了正确的起步，但尚未达到可进入后端阶段的成熟度。接下来必须严格围绕 PRD 完成前端，不应分散到后端开发。

现阶段最重要的不是扩展技术边界，而是把产品边界做实，把 PRD 定义的页面、链路、字段、口径和合规表达完整落到前端。只有当前端版本成为稳定、统一、可评审的产品表达之后，后端开发才有明确、低返工、可实施的输入基础。
