# 行业基金机会捕捉平台数据架构方案 V1

## 1. 文档信息

- 文档名称：行业基金机会捕捉平台数据架构方案 V1
- 文档版本：V1.0
- 文档角色：架构师数据方案说明
- 适用对象：产品经理、架构师、后端工程师、数据工程师、前端工程师、测试工程师、运维工程师
- 关联文档：
  - `E:\game\docs\prd-v1-industry-fund-opportunity.md`
  - `E:\game\docs\frontend-architecture-v1.md`
  - `E:\game\docs\architect-frontend-phase-note.md`
- 文档目标：在前端 V1 基本确定的前提下，给出适用于盘后数据采集与 T+1 展示的数据架构方案，支撑行业机会评分、基金筛选、基金对比和观察列表的后续工程化建设

## 2. 架构结论摘要

当前版本不建设实时流式数据平台，不引入毫秒级或分钟级行情链路。V1 到 V2 的数据架构采用“盘后批处理采集 + 标准化加工 + 结构化存储 + API 按页面聚合输出”的渐进式方案，核心判断如下：

1. 基金对比数据按 T+1 或交易日盘后更新即可，当前产品不要求盘中实时净值对比。
2. 行业机会评分按交易日盘后计算即可，前端首页和行业详情页展示最近一次成功更新的结果。
3. 数据链路优先保证稳定、可解释、可补偿、可审计，而不是极致实时性。
4. Windows 环境下优先采用单机可运维的分层部署方式：任务调度、采集程序、加工程序、API 服务、数据库和日志分离部署或按阶段合并部署。
5. V1 以“日更数据服务前端演示与准生产展示”为目标，V2 再逐步引入更多数据源、任务编排能力和更细粒度的数据质量治理。

## 3. 业务边界与数据时效要求

### 3.1 本阶段支持的数据能力

1. 交易日盘后采集基金基础信息、净值表现、规模、费率、波动率、回撤等基础指标。
2. 交易日盘后采集行业指数、主题指数、行业热度、资金流代理指标、事件标签等数据。
3. 基于盘后数据计算行业机会评分、行业标签、风险等级和基金对比字段。
4. 向前端提供首页、行业详情页、基金发现页、基金对比页、观察页所需的结构化接口。
5. 保留数据更新时间、口径说明、数据状态和风险说明字段。

### 3.2 本阶段明确不做的能力

1. 不做盘中实时行情流处理。
2. 不做实时 websocket 推送。
3. 不做高频分钟线、tick 级别存储和回放。
4. 不做自动交易、下单、账户同步和持仓计算。
5. 不做用户级个性化推荐引擎和复杂画像系统。

### 3.3 时效性要求

| 数据域 | 更新频率 | 目标完成时间 | 前端展示口径 |
| --- | --- | --- | --- |
| 基金基础资料 | 每日一次或按变更同步 | 交易日盘后 | 展示最近一次有效更新 |
| 基金收益/风险指标 | 每交易日一次 | T 日收盘后至 T+1 凌晨前 | T+1 展示 |
| 行业机会评分 | 每交易日一次 | T 日收盘后统一计算 | T+1 展示 |
| 行业事件与催化标签 | 每交易日一次 | 盘后汇总 | 展示最近一次处理结果 |
| 观察列表关联摘要 | 每交易日一次 | 评分完成后刷新 | 展示最近更新时间 |

结论：基金对比和行业机会评分在 V1 均按 T+1 或盘后更新处理，不提供实时盘中刷新承诺。

## 4. 总体数据架构

### 4.1 分层结构

推荐采用五层结构：

1. 数据源层
   - 外部基金数据源
   - 外部行业指数或主题数据源
   - 事件/资讯/政策标签来源
   - 手工维护映射表与产品配置表
2. 采集层
   - 按交易日盘后执行的抓取任务
   - 标准化原始落库
   - 采集状态记录
3. 加工层
   - 指标计算
   - 行业与基金映射
   - 机会评分计算
   - 页面聚合数据集生成
4. 服务层
   - 面向前端页面的只读 API
   - 数据更新时间、版本号、状态码输出
   - 聚合查询与筛选查询
5. 运维治理层
   - 任务调度
   - 失败重试
   - 日志与告警
   - 数据质量校验
   - 手工补跑与回滚

### 4.2 推荐技术形态

V1 推荐技术形态如下：

1. 采集与加工：Python 脚本或 Node.js 脚本均可，推荐 Python，更适合数据处理和批任务。
2. 调度：Windows Task Scheduler 作为 V1 默认调度器，V2 再引入 Airflow 或 Prefect 一类编排工具。
3. 数据库：PostgreSQL 作为主数据库，承载结构化结果表、任务表和质量校验表。
4. API 服务：优先继续采用 Next.js API Route 或独立 Node.js API 服务；若后端职责增加，再拆为独立服务。
5. 文件落盘：原始采集文件可落本地磁盘归档，格式推荐 CSV/JSON，用于追溯和补跑。

### 4.3 架构原则

1. 先结果表，后复杂平台：优先保障前端页面可稳定消费的结果数据表。
2. 先批处理，后准实时：当前业务不需要为实时性付出高复杂度成本。
3. 先可解释，后自动化：机会评分和标签必须能追溯来源。
4. 先可补偿，后高并发：V1 更重要的是出错后能快速恢复。
5. 页面接口不直接读取原始采集表，必须经过加工层或聚合层。

## 5. 数据域划分

### 5.1 主数据域

1. `fund_master`
   - 基金基础主数据
   - 包含基金代码、名称、基金类型、主题、基金公司、是否场内交易、费率、成立日等
2. `industry_master`
   - 行业或主题主数据
   - 包含行业标识、行业名称、展示名称、排序权重、风险提示模板等
3. `industry_fund_mapping`
   - 行业与基金映射关系
   - 支撑行业详情页和基金发现页的联动

### 5.2 时序指标域

1. `fund_daily_metrics`
   - 每日基金收益、波动率、回撤、规模、换手代理指标等
2. `industry_daily_metrics`
   - 每日行业表现、资金强度代理、估值位置、热度等
3. `industry_events_daily`
   - 每日行业事件标签、催化说明、事件日期与摘要

### 5.3 结果输出域

1. `industry_opportunity_daily`
   - 每交易日的行业机会评分结果
   - 包含机会评分、趋势分、资金分、估值分、风险标签、结论标签、摘要说明
2. `fund_compare_daily`
   - 每交易日基金对比结果快照
   - 包含收益、回撤、波动、规模、费率、持仓摘要、跟踪偏离说明
3. `homepage_snapshot_daily`
   - 首页展示所需聚合结果
4. `industry_detail_snapshot_daily`
   - 行业详情页聚合快照
5. `watchlist_change_summary_daily`
   - 观察列表所需的最近变化摘要

### 5.4 治理与运行域

1. `job_run_log`
   - 记录每个任务的开始时间、结束时间、状态、处理量、错误信息
2. `data_quality_result`
   - 记录各项质量规则的执行结果
3. `source_ingestion_audit`
   - 记录原始数据采集批次、源文件位置、行数、摘要哈希

## 6. 盘后数据采集方案

### 6.1 盘后任务链路

推荐按固定顺序执行盘后任务：

1. 交易日判断任务
   - 判断当天是否为交易日
   - 非交易日可跳过主任务，仅执行健康检查
2. 主数据同步任务
   - 更新基金基础资料和行业主数据
3. 行情与指标采集任务
   - 采集基金日级数据
   - 采集行业日级数据
   - 采集事件标签或资讯摘要
4. 标准化入库任务
   - 清洗字段、统一类型、去重后写入原始表或标准表
5. 指标加工任务
   - 计算收益率、波动率、回撤等基金指标
   - 计算行业趋势、资金强度、估值性价比、风险因子
6. 机会评分任务
   - 生成行业机会评分和标签
7. 页面快照生成任务
   - 生成首页、行业详情、基金对比、观察摘要等聚合结果
8. 发布标记任务
   - 将本次成功批次标记为当前前端可消费版本

### 6.2 推荐执行时间

在 A 股及公募基金业务语境下，V1 可采用以下默认时间窗口：

1. `18:30` 开始检查外部数据是否可用
2. `19:00 - 20:30` 执行基础采集与标准化入库
3. `20:30 - 21:30` 执行指标加工与评分计算
4. `21:30 - 22:00` 执行质量校验与结果发布
5. `22:00` 后前端 API 默认读取当日最新成功批次

若外部源存在延迟，允许将最终发布时间顺延至 `T+1 07:00` 前，但对前端仍统一标注为“最近一次有效更新”。

### 6.3 采集策略建议

1. 全量主数据低频同步，增量日指标高频按日同步。
2. 原始数据先落地再入库，保证可追溯。
3. 每个数据源单独记录抓取批次，避免一个源失败拖垮全部链路。
4. 同一指标若存在多个来源，V1 先固定主来源，不在本期做复杂多源自动仲裁。

## 7. 数据加工与评分计算

### 7.1 基金对比加工

基金对比模块按 T+1 结果生产，不要求盘中刷新。加工内容建议包括：

1. 近 1 月、3 月、6 月收益率
2. 最大回撤
3. 波动率
4. 规模区间
5. 费率
6. 成立时间
7. 前十大持仓摘要
8. 持仓集中度
9. 跟踪误差或风格偏离说明

输出结果直接写入 `fund_compare_daily`，前端对比页只读取该快照表或对应 API。

### 7.2 行业机会评分加工

行业机会评分也按盘后批处理计算。推荐采用“分项评分 + 标签输出”的方式，不做黑盒模型。

建议计算维度：

1. 趋势强度分
2. 资金强度分
3. 景气催化分
4. 估值性价比分
5. 风险因子分

最终输出：

1. 总机会评分
2. 机会标签
3. 风险等级
4. 一句话摘要
5. 各维度口径说明
6. 更新时间

### 7.3 页面快照加工

为降低前端接口复杂度，推荐在加工层就生成页面级快照：

1. 首页快照
   - 行业榜单
   - 热力图区块
   - 今日重点行业
   - 热门基金映射
2. 行业详情快照
   - 机会结论卡
   - 指标卡组
   - 趋势与资金图表序列
   - 事件时间线
   - 相关基金列表
3. 观察页摘要快照
   - 最近变化说明
   - 待跟踪提示

这样可以让前端 API 尽量走“读结果”的模式，而不是在请求时再进行复杂拼装。

## 8. 存储设计建议

### 8.1 主库选择

V1 推荐 PostgreSQL，原因如下：

1. 结构化数据能力成熟，适合结果表和日志表管理。
2. 支持事务、索引、视图和 JSON 字段，兼容页面快照和结构化结果混合存储。
3. Windows 下可稳定部署，后续迁移到 Linux 或云环境成本较低。

### 8.2 分层存储建议

1. 原始层
   - 存储原始抓取结果
   - 用于追溯和补跑
2. 标准层
   - 存储字段清洗、主键规范化后的标准数据
3. 服务层
   - 存储前端直接消费的聚合结果或快照

### 8.3 表设计原则

1. 时间快照类结果表以 `trade_date` 为核心分区或索引字段。
2. 所有结果表必须带 `updated_at`、`source_batch_id`、`data_version`。
3. 页面快照表可使用 JSON 字段承载前端聚合结构，但关键过滤字段仍应结构化存储。
4. 不建议 V1 一开始就做大量范式拆分，优先服务查询稳定性和开发效率。

## 9. 对外服务给前端的方式

### 9.1 服务原则

1. 前端读取的是“最近一次成功发布”的数据版本，而不是最新尚未校验的中间数据。
2. API 尽量按页面或业务能力聚合，减少前端请求数和拼装复杂度。
3. 每个响应都返回数据更新时间和数据状态。

### 9.2 推荐接口能力

在当前前端工程基础上，后续数据服务可继续沿用既有路由语义：

1. `/api/industries`
   - 返回首页行业榜单、热力图、重点行业卡片所需数据
2. `/api/industries/[id]`
   - 返回行业详情页所需快照
3. `/api/funds`
   - 返回基金发现页的筛选结果
4. `/api/compare`
   - 返回基金对比页快照
5. `/api/watchlist-summary`
   - 返回观察页所需变化摘要

### 9.3 响应字段建议

所有面向前端的响应建议增加以下公共字段：

1. `dataVersion`
2. `tradeDate`
3. `updatedAt`
4. `status`
5. `statusMessage`
6. `disclaimer`

### 9.4 发布策略

1. 只有当批次通过质量校验后，才切换前端读取的当前版本。
2. 若当日批次失败，则前端继续读取上一成功批次，并明确提示“当前展示最近一次成功更新数据”。
3. 前端不直接读取处理中状态的数据。

## 10. 失败补偿方案

### 10.1 失败场景分类

V1 重点处理以下失败场景：

1. 数据源连接失败
2. 单个源返回空数据
3. 字段结构变化导致解析失败
4. 部分表入库成功、部分表失败
5. 指标计算报错
6. 质量校验不通过
7. 发布阶段失败

### 10.2 补偿原则

1. 失败不覆盖上一成功版本。
2. 每个任务必须幂等，允许重跑。
3. 原始采集结果尽量保留，便于补跑。
4. 发布动作必须单独作为最后一步，避免半成品被前端读取。

### 10.3 具体补偿机制

1. 采集失败补偿
   - 单源失败时记录错误并支持独立重跑
   - 保留其他源已成功结果
2. 加工失败补偿
   - 以批次为单位重新执行加工
   - 不直接删除上一次成功快照
3. 发布失败补偿
   - 回退到上一成功版本
   - 记录“已完成加工未完成发布”的状态
4. 人工补偿
   - 提供按交易日、按任务名的手工重跑入口
   - 支持只重跑采集、只重跑加工、只重跑发布

### 10.4 前端兜底策略

1. 若当日无成功批次，则展示上一成功批次数据。
2. 若最近成功批次超过阈值，例如超过 2 个交易日未更新，则前端显示显著数据时效提示。
3. 若某个行业详情数据缺失，则页面进入局部降级，不影响全站可用性。

## 11. 数据质量校验方案

### 11.1 质量校验目标

V1 数据质量治理的目标不是构建复杂平台，而是保障前端展示不会出现明显错误、空值泛滥或口径异常。

### 11.2 规则分类

推荐至少覆盖四类规则：

1. 完整性校验
   - 关键表是否生成
   - 关键字段是否为空
   - 行业榜单和基金列表是否达到最小记录数
2. 唯一性校验
   - 基金代码是否重复
   - 行业主键是否重复
   - 同一交易日快照是否重复发布
3. 合理性校验
   - 收益率、波动率、回撤、费率等数值是否落在合理区间
   - 机会评分是否在预设范围
   - 风险等级与标签是否可映射
4. 关联性校验
   - 行业详情页中的相关基金是否都存在主数据
   - 首页重点行业是否都能打开详情页
   - 基金对比页目标基金是否都存在日快照

### 11.3 发布门禁建议

推荐将数据质量规则分为两级：

1. 阻断级
   - 主快照缺失
   - 关键字段为空
   - 结果表记录数明显不足
   - 同批次主键重复
2. 告警级
   - 个别行业事件为空
   - 个别基金持仓摘要缺失
   - 局部字段沿用上一日结果

阻断级失败时不发布新版本；告警级失败时允许发布，但必须记录并提示运维或人工复核。

### 11.4 质量结果留痕

每次批处理结束后都应写入 `data_quality_result`，至少记录：

1. 批次号
2. 规则名称
3. 规则级别
4. 校验结果
5. 失败样本
6. 执行时间

## 12. Windows 环境部署建议

### 12.1 V1 部署目标

用户明确在 Windows 上部署前后端，因此 V1 推荐以 Windows Server 或 Windows 11 专用服务器为基础环境，优先选择易于本地运维的单机方案。

### 12.2 推荐部署形态

#### 方案 A：V1 单机部署

适用于早期验证与低并发场景：

1. Windows 主机安装 PostgreSQL
2. Windows 主机安装 Node.js，用于前端与 API 服务
3. Windows 主机安装 Python，用于采集和加工任务
4. 使用 Windows Task Scheduler 调度盘后任务
5. 使用 NSSM 或 PM2 for Windows 托管 API/前端服务进程

#### 方案 B：V1.5 分进程部署

适用于需要更稳健运行的场景：

1. 一组常驻服务进程负责前端/API
2. 一组批处理脚本负责盘后采集与加工
3. 数据库独立实例
4. 日志目录、归档目录、配置目录独立管理

### 12.3 目录建议

Windows 服务器建议统一目录，例如：

1. `E:\game`：应用代码目录
2. `E:\game\runtime\logs`：日志目录
3. `E:\game\runtime\data-archive`：原始数据归档目录
4. `E:\game\runtime\config`：运行配置目录
5. `E:\game\runtime\scripts`：调度脚本目录

### 12.4 调度建议

V1 使用 Windows Task Scheduler 即可，推荐拆分为多个任务：

1. `market-calendar-check`
2. `fund-ingestion-daily`
3. `industry-ingestion-daily`
4. `data-transform-daily`
5. `opportunity-score-daily`
6. `snapshot-publish-daily`
7. `data-quality-report-daily`

每个任务单独记录日志，便于失败定位和手工补跑。

### 12.5 服务托管建议

1. 前端/API 服务建议使用 NSSM 或 PM2 托管，支持自启动与异常重启。
2. 采集和加工任务建议通过 PowerShell 启动脚本统一封装环境变量和日志输出。
3. PostgreSQL 建议配置定时备份与日志轮转。

### 12.6 安全与运维建议

1. 配置文件与敏感凭据不要写死在代码库，使用环境变量或独立配置文件。
2. 任务执行账户与服务运行账户分离。
3. 日志目录、归档目录和数据库备份目录定期清理与归档。
4. 为关键任务增加邮件或企业微信类告警能力，V1 可先实现邮件告警。

## 13. 从 V1 到 V2 的渐进式演进

### 13.1 V1

目标：支撑前端 V1 稳定读取盘后结果

1. Windows 单机部署
2. 盘后批处理任务
3. PostgreSQL 主库
4. API 读取结果快照
5. 手工补跑和基础质量校验

### 13.2 V1.5

目标：提升稳定性与运维效率

1. 增加任务依赖编排
2. 增加更细的质量规则
3. 增加数据源冗余或备用源
4. 增加接口缓存
5. 增加更明确的告警和日报

### 13.3 V2

目标：从“页面数据服务”升级为“投研型数据底座”

1. 引入调度编排平台，例如 Airflow 或 Prefect
2. 引入更规范的数据分层与版本管理
3. 增加多源事件归因和 AI 摘要加工
4. 增加更丰富的行业轮动与横向比较指标
5. 视业务需要再评估准实时能力，但不默认升级到流式架构

## 14. 推荐落地顺序

为了不偏离当前前端已确定的 PRD 范围，建议后续按以下顺序推进：

1. 先定义数据库表与批次模型，只覆盖当前前端已使用的数据契约。
2. 再实现盘后采集脚本和加工脚本，优先产出首页、行业详情、基金发现、基金对比所需结果表。
3. 然后补齐发布、失败补偿和质量校验。
4. 最后才让前端从 Mock API 平滑切换到真实结果 API。

## 15. 关键结论

1. 当前产品的数据架构应明确采用盘后批处理模式，不建设实时流式架构。
2. 基金对比和行业机会评分都按 T+1 或交易日盘后更新即可，前端只消费最近一次成功发布的数据版本。
3. V1 最合适的落地路径是“Windows 调度任务 + Python 批处理 + PostgreSQL 存储 + Node/Next API 输出”。
4. 数据链路必须具备失败不覆盖旧版本、任务可重跑、质量校验后发布的机制。
5. 页面接口应直接读取加工后的快照或结果表，而不是读取原始采集数据。
6. V2 的演进方向应是提升调度编排、数据治理和多源加工能力，而不是一开始就引入复杂流式平台。
